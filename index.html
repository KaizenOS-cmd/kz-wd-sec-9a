01
How to Use This SOP
Read before executing any security configuration

Workday security is foundational — mistakes here affect every worker, every process, and every audit. This SOP provides a structured, repeatable approach to security group design and domain policy assignment for both greenfield builds and post-go-live changes.

🔴 Critical
Security changes in Workday are not immediately reversible without an Activate Pending Security Policy Changes step. This creates a window to review — but once activated, incorrect changes can lock users out of critical tasks or expose sensitive data. Always test in sandbox before production. Always have a named approver for every security change.
Intended Use Cases
  • New Workday implementation — initial security model design and configuration
  • Post-go-live security group additions for new roles, orgs, or acquired entities
  • Annual access reviews and security model remediation
  • Consultant handover — documenting the as-built security architecture for client teams
  • Audit preparation — producing evidence of least-privilege access controls
✅ Tip
Duplicate this document per security group or per role family. Name files using your org's convention (e.g., SOP_SG_HR_Partner_v1.0_2026.html). Treat each file as the living record of that group's design intent — invaluable during audits and Works Council reviews.
02
Security Architecture Primer
Core concepts before you configure anything

Workday security operates on a layered model. Understanding all three layers — Security Groups, Domain Security Policies, and Business Process Security Policies — is required before touching any configuration.

Security Groups

Define who can access what. Groups are collections of users (via role, org membership, or explicit assignment). All access flows through group membership — never directly to individual users.

Domain Security Policies

Define what data is accessible. Domains are functional areas (e.g., Worker Data, Compensation, Benefits). Each domain has Get/Put permissions assigned per security group.

Business Process Policies

Define what actions are permitted. Each business process (e.g., Hire, Terminate, Change Job) has initiate, approve, and view permissions per security group.

Constrained vs. Unconstrained

Security groups can be constrained (access limited to workers in a specific org hierarchy) or unconstrained (global access). Always default to constrained for HR roles.

Security Group Types — Decision Guide
Group TypeMembership BasisBest ForConstraint Support
Role-Based Assigned via staffing role on a position or supervisory org HR Partners, Recruiters, Managers Constrained
User-Based Explicit user list — manually maintained Admins, Super Users, temporary access Unconstrained
Job-Based Automatically assigned by job profile or job family Self-service access (ESS/MSS) Constrained
Intersection Combination of two existing groups Layered access (e.g., HR Partner within specific org) Constrained
Aggregation Union of multiple security groups Simplifying policy assignments across groups Varies
💡 Rationale
Most organisations over-rely on User-Based groups because they're easy to set up. This creates maintenance debt — every new hire in an admin role requires a manual update. Design toward Role-Based groups wherever possible. Reserve User-Based for genuine exceptions (system accounts, break-glass access).

04
Security Group Configuration — Step-by-Step
Creating, configuring, and scoping security groups in Workday
Part A — Creating a Role-Based Security Group (Most Common)
1
Navigate to Create Security Group
Search Create Security Group in the Workday search bar. Select the group type: Role-Based Security Group. Enter the group name per your naming convention. Add a description — be specific about the intended role and access scope. This description is visible to auditors and future admins.
✅ Tip
Write descriptions in plain language that a non-technical HR leader can understand. Example: "Grants EMEA HR Partners access to view and edit worker data, compensation, and absence records for workers in their assigned supervisory org hierarchy."
2
Assign the Workday Role
In the Workday Role field, select or create the staffing role that drives group membership (e.g., HR Partner, Recruiter, Benefits Partner). If the role does not exist, navigate to Create Role first and define it — roles are the mechanism that connects workers to security groups automatically when assigned to a position.
📋 Note
Role assignment happens at the Supervisory Org level. A worker gains security group membership when they are assigned the matching role on a supervisory org — and loses it automatically when the role is removed or they leave. This is the self-maintaining nature of role-based groups.
3
Configure the Constrained Scope
Under Permitted Security Groups to Administer, define whether this group is org-constrained (access limited to the org hierarchy where the role is assigned) or unrestricted. For HR Partner groups, always select org-constrained. Confirm the constraint applies to all downstream worker data access — not just the top-level org.
🔴 Critical
Selecting Unrestricted grants the security group access to ALL workers across the entire tenant. This is appropriate only for Global HRIS Admin and senior platform owners. For every other role, constrained access is mandatory.
4
Save & Document
Save the security group. Copy the Workday system ID from the URL and record it in your Domain Access Matrix and Operator Notes (Section 09). At this stage the group exists but has no permissions — domain and BP policy assignment follows in Section 05.
Part B — Creating a User-Based Security Group
1
Create & Name the Group
Search Create Security Group → select User-Based Security Group. Apply your naming convention. In the description, document the specific reason this is user-based rather than role-based, and the review cadence (e.g., "Quarterly access review required — see Change Log").
💡 Rationale
Forcing yourself to document why a group is user-based creates accountability. It also flags the group as requiring active maintenance — unlike role-based groups, user-based groups do not self-update when workers change roles or leave.
2
Add Members
Navigate to the group → Maintain Members for Security Group. Add users individually by searching for their Workday accounts. Confirm each user's access is still required before adding — do not copy membership from a previous group without verification.
⚠️ Warning
User-Based group membership does NOT automatically remove terminated workers. Schedule a quarterly review task to audit membership — terminated workers retaining group membership is a common SOX and GDPR audit finding.
Part C — Creating an Intersection Security Group
1
Identify the Two Parent Groups
Intersection groups restrict access to workers who appear in both parent groups. Common use case: an HR Partner who should only see Compensation data for workers in their supervisory org. Parent Group 1: the HR Partner role-based group. Parent Group 2: a compensation data group.
2
Create the Intersection Group
Search Create Security Group → select Intersection Security Group. Select the two parent groups. Name and describe the intersection clearly — include both parent group names in the description. Save. Assign domain policies for the intersection-specific permissions in Section 05.
📋 Note
Intersection groups do not inherit permissions from parent groups — they must be independently assigned domain and BP policies. This is a common configuration error: creating the intersection but forgetting to assign it to any policies.
05
Domain & Business Process Policy Assignment
Granting permissions to security groups via domain and BP policies

Creating a security group grants no access by itself. Access is granted through Domain Security Policies (data access) and Business Process Security Policies (action access). Both must be configured — and both require the Activate Pending Security Policy Changes step before taking effect.

Part A — Domain Security Policy Assignment
1
Navigate to the Domain Security Policy
Search for the domain by name (e.g., Worker Data: Personal Data, Compensation: Salary). Use the Domain Security Policy task. Browse by functional area or use the search. Workday has hundreds of domains — use your Domain Access Matrix to identify exactly which domains are in scope.
✅ Tip
Use the View Domain Security Policies for Security Group report to get a full list of domains currently assigned to a group. Run this before and after changes as a before/after audit record.
2
Add the Security Group with Correct Permission Type
Within the Domain Security Policy, navigate to Edit. Add your security group to the relevant permission tier:

Get — View/read access only
Put — Create or edit access
Get & Put — Full read and write

Apply the minimum permission level required. If the role only needs to view compensation data, assign Get only — never default to Get & Put.
🔴 Critical
Put permissions on sensitive domains (Compensation, Benefits, Banking) must be explicitly justified and approved. Assigning Put access to the wrong group is one of the most common security findings in Workday audits — and one of the hardest to remediate retroactively.
3
Save & Repeat for All In-Scope Domains
Save the domain policy. Repeat for each domain in your Domain Access Matrix. Work through domains systematically by functional area — Worker Data, Compensation, Benefits, Absence, Recruiting — to avoid gaps. Track completion in your matrix.
Part B — Business Process Security Policy Assignment
1
Navigate to the Business Process Security Policy
Search Business Process Security Policy and select the relevant process (e.g., Hire Employee, Change Job, Terminate Employee). Each business process has its own policy. Identify all in-scope BPs from your BP Policy Matrix before beginning.
2
Assign Initiate, Approve, or View Permissions
Within the BP Security Policy, assign the security group to the appropriate permission type:

Initiate — Can start the business process
Approve — Can approve steps in the process
View — Can see process events but not act on them

For HR Partner groups, typical assignment is Initiate + View on most HR BPs, with Approve reserved for managers and HR leadership.
⚠️ Warning
Initiate permission on Terminate Employee or Change Job must be constrained. An unconstrained Initiate on Terminate Employee effectively allows the user to terminate any worker in the system. Always verify constraint applies before activating.
3
Activate Pending Security Policy Changes
Once all domain and BP policy changes are saved, navigate to Activate Pending Security Policy Changes. Enter a comment describing the changes being activated (this is your audit trail). Review the pending changes list carefully — confirm only your intended changes are listed. Click OK to activate.
🔴 Critical
Activating security policy changes is immediate and affects all users in the impacted security groups. In production, always activate during a low-traffic window. In sandbox, activate freely to test. Never activate production security changes without a second reviewer confirming the pending changes list.
06
Validation, Testing & Activation
Confirming access works as designed before sign-off
View As — Persona Testing

Workday's View As feature allows you to simulate the system as a specific user or security group. Use this to validate access before activating in production.

1
Use View As for Each Role Persona
Navigate to View Workday Account → select a test user assigned to the security group. Click Switch to This Account. Navigate through the expected workflows for that persona. Confirm: menus display correctly, restricted data is hidden, permitted tasks are accessible.
2
Test Constraint Boundaries
For constrained groups, attempt to access a worker outside the test user's org scope. Confirm that the worker's record is not accessible or returns a security error. This is the most important test for HR Partner and Manager groups — constraint failures expose PII across org boundaries.
✅ Tip
Create two test workers — one inside the org scope, one outside. Test both in the same session. Document the results with screenshots for your audit record.
3
Test Business Process Initiation
As the test persona, attempt to initiate each Business Process in scope (e.g., Change Job, Request Absence). Confirm the task is available and the workflow routes correctly. Attempt a BP outside scope — confirm it is inaccessible or returns a security error.
Validation Checklist
  • View As test completed for each security group personaAt least one test user per group; document results
  • Constraint boundary tested — out-of-scope workers inaccessible
  • All in-scope Business Processes initiate correctly
  • Out-of-scope Business Processes return security error as expected
  • Sensitive domains (Compensation, Banking) confirmed inaccessible to non-permitted groups
  • Workday reports confirm correct group membership for test users
  • UAT sign-off obtained from HR lead and IT Security reviewer

08
Rollback & Risk Notes
Reversing incorrect security changes and managing common risks
1
Identify the Incorrect Change
Navigate to Security Audit Report or View Security Policy Change History. Identify the specific domain policy or BP policy that was incorrectly changed. Note the security group affected, the permission type, and the activation timestamp.
2
Remove the Incorrect Permission
Navigate to the domain or BP policy. Edit — remove the security group from the incorrect permission tier or remove the group entirely if it should not have been added. Save the change. At this point, the correction is pending — it has not yet taken effect.
⚠️ Warning
Between activation of an incorrect change and activation of the correction, users may have accessed data or initiated BPs they should not have. If sensitive data (compensation, banking, health) was accessed, escalate to your DPO — this may constitute a reportable data incident.
3
Activate the Correction
Run Activate Pending Security Policy Changes. Add a detailed comment referencing the incident (e.g., "Correction — SG_RB_HRPartner_EMEA incorrectly granted Put on Compensation:Salary on [date]. Removed per incident [REF]."). This comment is your permanent audit record. Activate.
4
Validate & Document
Re-run View As testing to confirm the incorrect access is no longer available. Document the incident, root cause, correction, and any data access that occurred during the window in the Change Log (Section 10) and your incident management system.
